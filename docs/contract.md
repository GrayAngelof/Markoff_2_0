## 🏛️ **РАЗДЕЛ 1: ПРОЦЕСС РАЗРАБОТКИ**

### **Статья 1.1. Обсуждение перед кодом**
> ПЕРЕД НАПИСАНИЕМ КОДА ОБСУДИ И ЗАПОЛНИ СПЕЦИФИКАЦИЮ
> ЕСЛИ СПЕЦИФИКАЦИЯ УЖЕ ЕСТЬ, ТО СТРОГО ПРИДЕРЖИВАТЬСЯ СПЕЦИФИКАЦИИ!
- Каждый новый модуль начинается с контракта (спецификации)
- Без утвержденной спецификации код не пишется
- Спецификация отвечает на вопросы:
  - Зачем? (проблема)
  - Где? (место в иерархии)
  - Что? (границы ответственности)
  - Кто? (потребители)
  - Как? (ключевые решения)

## 🏛️ **РАЗДЕЛ 2: КАЧЕСТВО КОДА**

### **Статья 2.1. Читаемость**
> КОММЕНТАРИИ — ЧИТАЕМОСТЬ ВЫШЕ КОМПАКТНОСТИ

- Код читается чаще, чем пишется
- Лучше 10 строк понятных, чем 5 строк "гениальных"
- Комментарии объясняют **почему**, а не **что** (что видно из кода)
- Сложная логика обязана иметь комментарий

### **Статья 2.2. Именование**
> ИМЕНА — ОДНОЗНАЧНОСТЬ

- Имя переменной/метода/класса отвечает на вопрос "что это?"
- Нет аббревиатур без расшифровки
- Длина имени пропорциональна области видимости
- Глаголы для методов, существительные для классов

### **Статья 2.3. Документация**
> ДОКУМЕНТАЦИЯ — НЕ РАДИ ВИТРИНЫ, ПОНИМАНИЕ ВАЖНО

- Каждый файл начинается с комментария вида: # {project_root}/{layer}/{sub_layer…}/{filename}.py
- Докстринги есть у каждого публичного API
- Докстринг отвечает: что делает, параметры, возврат, исключения
- Примеры использования для сложных случаев
- Документация пишется для людей, не для машин

## 🏛️ **РАЗДЕЛ 3: АРХИТЕКТУРА**

### **Статья 3.1. CORE — СВЯЩЕННАЯ КОРОВА**
- Core **никогда не меняется**
- Если хочешь что-то поменять в core — **ты ошибся**
- Core только экспортирует типы и контракты
- Core не имеет зависимостей (кроме стандартной библиотеки)
- Исключений нет

### **Статья 3.2. Строгая иерархия слоев**
> СТРОГАЯ ЗАВИСИМОСТЬ СЛОЕВ, ТОЛЬКО СВЕРХУ ВНИЗ

```
core → models → data → services → projections → controllers → ui
```

- Нижний слой не знает о верхнем
- Нарушение = архитектурное преступление
- Циклические зависимости запрещены

### **Статья 3.3. Интеграция и пакеты**
> ИНТЕГРАЦИЯ И ПАКЕТЫ

- Каждый слой — отдельный пакет (папка с `__init__.py`)
- Публичное API слоя — через `__init__.py`
- Внутренности — в подпапках (приватно)
- Никто не импортирует внутренности напрямую

### **Статья 3.4. Консолидация констант**
> КОНСОЛИДАЦИЯ КОНСТАНТ

- Все константы в одном месте (core/constants.py)
- Нет магических строк в коде
- События — только через core/events.py
- Исключение: локальные константы в модуле (с комментарием)

### **Статья 3.5. Потоки данных**
> СТРОГОЕ РАЗДЕЛЕНИЕ ПОТОКОВ ДАННЫХ

- Данные текут в одном направлении
- Нет "боковых" потоков
- Изменение данных только через владельца
- Read-only доступ везде

## 🏛️ **РАЗДЕЛ 4: ПРАКТИКИ КОДИРОВАНИЯ**

### **Статья 4.1. YAGNI**
> You Ain't Gonna Need It

- Пишем код **только для реальных, существующих задач**
- Нет "а вдруг пригодится"
- Нет "на всякий случай"
- Если код не используется — **удаляем немедленно**

### **Статья 4.2. Никакой магии**
- Все зависимости — явные
- Нет скрытого поведения
- Нет импортов "из воздуха"
- Если что-то не очевидно — **это баг**

### **Статья 4.3. Единая ответственность**
- Класс делает **одно дело**
- Метод делает **одно действие**
- Если "и то, и сё" — **дели**
- Признак нарушения: трудно назвать метод

### **Статья 4.4. Типизация**
- Везде аннотации типов
- Нет `Any` без крайней необходимости
- IDE должна помогать, а не гадать
- mypy проходит без ошибок

### **Статья 4.5. Логирование**
> ВСЕГДА ИСПОЛЬЗОВАТЬ НАШ ЛОГГЕР ВО ВСЕМ ЕГО ФУНКЦИОНАЛЕ

```python
from utils.logger import get_logger
log = get_logger(__name__)

Категории и их уровни по умолчанию
_default_levels = {
  # КРИТИЧЕСКИЕ ОШИБКИ (ERROR)
    "error": LogLevel.ERROR,        # Ошибки (исключения, сбои)
    
  # ПРЕДУПРЕЖДЕНИЯ (WARNING)
    "warning": LogLevel.WARNING,    # Предупреждения (некритичные проблемы)
    "db": LogLevel.WARNING,         # База данных (медленные запросы, ошибки)
    
  # ЖИЗНЕННЫЙ ЦИКЛ ПРИЛОЖЕНИЯ (INFO)
    "startup": LogLevel.INFO,       # Запуск приложения (инициализация)
    "shutdown": LogLevel.INFO,      # Остановка приложения (освобождение ресурсов)
    
  # ИНФРАСТРУКТУРА (INFO)
    "system": LogLevel.INFO,        # Система (конфигурация, окружение)
    "api": LogLevel.INFO,           # API запросы (входящие/исходящие, статус коды)
    "link": LogLevel.INFO,          # Ссылки и соединения (подключения к сервисам)
    
  # БИЗНЕС-ЛОГИКА (INFO)
    "data": LogLevel.INFO,          # Данные (загрузка, сохранение, трансформация)
    "success": LogLevel.INFO,       # Успешные операции
    "info": LogLevel.INFO,          # Общая информация
    
  # ПРОИЗВОДИТЕЛЬНОСТЬ
    "performance": LogLevel.INFO,   # Метрики производительности
      with log.measure_time("..."):   # Логирует время выполнения
    
  # ОТЛАДКА (DEBUG)
    "debug": LogLevel.DEBUG,        # Отладка (подробная диагностика)
    "cache": LogLevel.DEBUG,        # Кэш (попадания/промахи, инвалидация)
}

Иконки для разных уровней и категорий
```

- Нет print()
- Нет случайного логирования
- Уровни используются осмысленно и в соответствии с контекстом

### **Статья 4.6. Иммутабельность**
- Данные не меняются после создания
- Если нужно изменить — создай новое
- Особенно для DTO и кэша
- `frozen=True` для dataclass'ов

### **Статья 4.7. Тесты**
- Код без тестов = сломанный код
- Тесты пишутся **до или вместе** с кодом
- Регрессия недопустима
- Покрытие > 80%

## 🏛️ **РАЗДЕЛ 5: КОММУНИКАЦИЯ**

### **Статья 5.1. События**
- Компоненты общаются через EventBus
- Нет прямых вызовов между слоями
- События типизированы (не `Dict[str, Any]`)
- Все события в core/events.py

### **Статья 5.2. Явное лучше неявного**
- Параметры явные, а не через словари
- Константы, а не магические строки
- Читаемость важнее краткости
- Если непонятно с первого взгляда — переделывай
