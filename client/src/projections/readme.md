## Анализ слоя: **projections** (слой проекций данных для UI)

### Краткое описание слоя

**Назначение** – преобразовывать данные из сервисного слоя и репозиториев в структуры, удобные для отображения в пользовательском интерфейсе. Слой `projections` создаёт **представления** (view models) без привязки к конкретному UI-фреймворку, обеспечивая единообразный доступ к данным для контроллеров и UI.

**Что делает:**
- Строит иерархическое дерево узлов (`TreeNode`) для отображения в дереве (TreeView)
- Формирует отображаемые имена (например, "Этаж 3", "Подвал 1", "Корпус А (5)")
- Предоставляет методы навигации по узлам (`find_child_by_id`, `row`, `child_at`)
- Скрывает детали хранения (ссылки на родителя, детей) от UI
- Сортирует дочерние узлы по имени (алфавитный порядок)

**Что не должен делать:**
- Содержать бизнес-логику, загрузку данных, валидацию (это в `services`)
- Обращаться напрямую к API, БД или графу (`data`) – только через репозитории, но лучше через сервисы
- Содержать UI-специфичный код (Qt-виджеты, реакции на клики) – это в `controllers` и `ui`
- Импортировать что-либо из `controllers`, `ui`

**Зависимости:** от `core` (типы, идентификаторы), от `models` (DTO), от `data` (репозитории). Никаких зависимостей от `services` или выше.

---

### Внутренние классы (кратко)

| Класс | Модуль | Назначение |
|-------|--------|------------|
| `TreeNode` | `tree_node.py` | Узел дерева для отображения. Хранит исходные данные (`data`), тип, отображаемое имя, флаг `has_children`, ссылки на родителя и детей. Предоставляет методы навигации (`find_child_by_id`, `row`, `child_at`) и управления детьми (`append_child`, `remove_child`). Иммутабелен по отношению к данным, но мутабелен по связям (дети добавляются). |
| `TreeProjection` | `tree.py` | Проекция дерева. Строит корневые узлы (`get_root_nodes`) из репозитория комплексов. Создаёт дочерние узлы из загруженных payload (`build_children_from_payload`). Форматирует отображаемые имена (этажи, подвалы, счётчики детей). |

---

### Внутренние импорты (только между модулями projections, core, models, data)

- `tree_node.py` → импортирует `NodeIdentifier`, `NodeType` из `core.types`; `utils.logger`
- `tree.py` → импортирует `NodeType` из `core.types`; репозитории из `src.data.repositories` (`ComplexRepository`, `BuildingRepository`, `FloorRepository`, `RoomRepository`); `TreeNode` из текущего пакета; `utils.logger`

**Никаких импортов из `services`, `controllers`, `ui`.**

### Экспортируемые методы / классы для вышестоящих слоёв

Публичное API слоя `projections` доступно через `from src.projections import ...` (согласно `projections/__init__.py`):

**Классы:**
- `TreeProjection`
- `TreeNode`

**Методы `TreeProjection`:**
- `get_root_nodes() -> List[TreeNode]` – возвращает список корневых узлов (комплексов) для начального отображения дерева.
- `build_children_from_payload(payload: List[Any], child_type: NodeType, parent_node: TreeNode) -> List[TreeNode]` – создаёт дочерние узлы из загруженных данных (например, списка Building DTO) и привязывает их к родителю.

**Методы `TreeNode` (публичные, для UI и контроллеров):**
- Свойства: `id: int`, `type: str`, `name: str`, `has_children: bool`
- Навигация: `children -> List[TreeNode]`, `parent -> Optional[TreeNode]`, `child_at(row: int) -> Optional[TreeNode]`, `child_count() -> int`, `row() -> int`
- Поиск: `find_child_by_id(node_type: NodeType, node_id: int) -> Optional[TreeNode]`
- Идентификация: `get_id() -> int`, `get_identifier() -> NodeIdentifier`
- Управление детьми (используется при построении): `append_child(child)`, `add_children(children)`, `remove_child(child) -> bool`, `remove_all_children()`
- Строковое представление: `__repr__`, `__str__`

### Итоговое заключение: принципы работы со слоем `projections`

1. **Зависимость только от `core`, `models`, `data`** – слой `projections` не должен импортировать `services`, `controllers`, `ui`. Он может использовать репозитории напрямую (как `TreeProjection`), но лучше получать уже загруженные данные через сервисы (чтобы не дублировать логику загрузки). В текущей реализации `TreeProjection.get_root_nodes()` сам вызывает `complex_repo.get_all()`, что допустимо, так как репозитории – часть слоя `data`.

2. **`TreeNode` – это view model, а не DTO** – он содержит ссылки на родителя и детей, что удобно для UI-компонентов (например, QTreeView). Однако **не храните в `TreeNode` бизнес-логику** – только навигацию и отображение. Исходные данные доступны через свойство `data` (сами DTO из `models`).

3. **Строительство дерева** – используйте `TreeProjection` для создания корневых узлов и для преобразования загруженных списков детей в `TreeNode`. После получения списка узлов от `build_children_from_payload` не забудьте добавить их в родителя через `parent_node.add_children(nodes)` (это делает вызывающий код, обычно в контроллере).

4. **Форматирование отображения** – всё форматирование имён (этажи, подвалы, счётчики) должно быть в `TreeProjection._get_display_name()`. Не разбрасывайте логику форматирования по UI-слою.

5. **Сортировка** – дочерние узлы сортируются по имени (регистронезависимо) внутри `build_children_from_payload`. Это единое место, гарантирующее порядок во всём приложении.

6. **Не мутируйте `TreeNode` из UI** – все изменения структуры дерева (добавление/удаление детей) должны происходить через контроллеры, которые получают новые данные от сервисов и перестраивают проекцию. Сам `TreeNode` не должен содержать методов загрузки данных.

7. **Расширение** – при добавлении нового типа узла:
   - Добавьте соответствующий репозиторий в `TreeProjection` (если нужен для корневых узлов)
   - Дополните `_get_display_name` для форматирования
   - В `build_children_from_payload` добавьте определение `has_children` для нового типа (если он может иметь детей)
   - Обновите `TreeNode` (обычно не требуется, так как он универсален)

Слой `projections` является **мостиком между данными и представлением**. Он не знает, как именно данные будут отображаться (Qt, веб, консоль), но предоставляет структуры, которые легко использовать в любом UI. Контроллеры должны использовать проекции для получения готовых к отображению объектов.
