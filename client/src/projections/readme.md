# ============================================
# СПЕЦИФИКАЦИЯ: Проекции (Projections)
# ============================================

## 1. НАЗНАЧЕНИЕ

Проекции — это **слой преобразования данных**, который принимает на вход модели данных из репозиториев/графа и возвращает View Models — структуры, подготовленные для отображения в UI. Проекции решают задачу изоляции UI от формата хранения данных: UI не знает, как устроены Complex, Building, Floor, Room — он работает только с View Models.

**Ключевая идея:** Проекции — это **чистые преобразователи без состояния**. Они не подписываются на события, не кэшируют, не принимают решений. Они только читают данные и возвращают готовую структуру для UI.

## 3. ЗА ЧТО ОТВЕЧАЕТ

### **Отвечает за:**
- Преобразование моделей данных (Complex, Building, Floor, Room, Counterparty, ResponsiblePerson) в View Models
- Агрегацию данных из нескольких репозиториев (например, статистика по корпусу требует данных из building_repo, floor_repo, room_repo)
- Подсчет производных показателей (occupancy_rate, total_area, room_types)
- Формирование вложенных структур (группировка контактов по категориям)

### **НЕ отвечает за:**
- Загрузку данных (это DataLoader)
- Хранение данных (это EntityGraph)
- Решение, когда обновлять UI (это контроллер)
- Кэширование результатов (это ответственность контроллера, если нужно)
- Подписку на события (это контроллер)
- Бизнес-логику загрузки ("если у корпуса есть владелец, загрузить его") — это DataLoader

---

## 4. КТО ИСПОЛЬЗУЕТ

### **Создают (источники):**
- **Проекции не создаются — они существуют как сервисы**
- **Контроллеры** — вызывают проекции, передавая идентификатор узла

### **Вызывают:**
- **DetailsController** — при `_rebuild()` вызывает все проекции для текущего узла
- **Другие контроллеры** — по мере необходимости (например, для обновления списков)

### **Вызывают внутри себя:**
- Репозитории (`ComplexRepository`, `BuildingRepository`, `FloorRepository`, `RoomRepository`, `CounterpartyRepository`, `ResponsiblePersonRepository`)
- EntityGraph (опционально, через репозитории)

### **НЕ используют:**
- UI — UI не вызывает проекции напрямую (только через события от контроллера)
- DataLoader — DataLoader не знает о проекциях
- Слой data — репозитории не знают о проекциях

---

## 5. КЛЮЧЕВЫЕ ПОНЯТИЯ

| Термин | Значение |
|--------|----------|
| **Чистое преобразование** | Функция, которая получает входные данные и возвращает результат без побочных эффектов |
| **Агрегация** | Сбор данных из нескольких источников (репозиториев) для формирования одной View Model |
| **Производный показатель** | Значение, вычисляемое на основе других данных (например, occupancy_rate = occupied_rooms / total_rooms * 100) |
| **Группировка** | Объединение данных по категориям (например, контакты по типам вопросов) |
| **Без состояния** | Проекция не хранит данные между вызовами — каждый вызов начинается с чистого листа |

## 6. ОГРАНИЧЕНИЯ (ВАЖНО!)

### **Запрещено:**
1. **Хранить состояние (кэш) внутри проекции**
   - ❌ `self._cache: Dict = {}`
   - ✅ Контроллер решает, нужно ли кэшировать

2. **Подписываться на события**
   - ❌ `self._bus.subscribe(DataLoaded, self._on_data)`
   - ✅ Только контроллеры подписываются и вызывают проекции

3. **Принимать решения о загрузке данных**
   - ❌ `if not data: self._loader.load(...)`
   - ✅ Если данных нет, возвращаем View Model с нулевыми значениями

4. **Возвращать None**
   - ❌ `return None` если данных нет
   - ✅ Всегда возвращаем View Model (с нулевыми/пустыми значениями)

5. **Содержать бизнес-логику**
   - ❌ "если owner_id есть, загрузить владельца"
   - ✅ Только преобразование уже загруженных данных

6. **Знать о существовании EventBus**
   - ❌ `self._bus.emit(...)`
   - ✅ Проекция ничего не эмитит — это делает контроллер

7. **Обращаться к UI или контроллерам**
   - ❌ Импорт `controllers` или `ui`
   - ✅ Только `models`, `data.repositories`, `view_models`

## 7. ПРИМЕРЫ СТРУКТУРЫ (что внутри каждой проекции)

### **StatisticsProjection**

| Метод | Вход | Выход | Что делает |
|-------|------|-------|------------|
| `build_for_complex(complex_id)` | int | `ComplexStatisticsVM` | Получает корпуса → этажи → помещения, считает агрегаты |
| `build_for_building(building_id)` | int | `BuildingStatisticsVM` | Получает этажи → помещения, считает агрегаты |
| `build_for_floor(floor_id)` | int | `FloorStatisticsVM` | Получает помещения, считает агрегаты |

**Зависимости:** ComplexRepository, BuildingRepository, FloorRepository, RoomRepository

---

### **ContactsProjection**

| Метод | Вход | Выход | Что делает |
|-------|------|-------|------------|
| `build_for_complex(complex_id)` | int | `ContactsVM` | Собирает владельцев комплекса + их контакты |
| `build_for_building(building_id)` | int | `ContactsVM` | Собирает владельца корпуса + его контакты |

**Зависимости:** BuildingRepository, CounterpartyRepository, ResponsiblePersonRepository

---

### **SensorsProjection**

| Метод | Вход | Выход | Что делает |
|-------|------|-------|------------|
| `build_for_complex(complex_id)` | int | `SensorsVM` | Собирает все датчики всех помещений комплекса |
| `build_for_building(building_id)` | int | `SensorsVM` | Собирает датчики всех помещений корпуса |
| `build_for_floor(floor_id)` | int | `SensorsVM` | Собирает датчики помещений этажа |
| `build_for_room(room_id)` | int | `SensorsVM` | Собирает датчики помещения |

**Зависимости:** RoomRepository, SensorRepository (если есть)

---

### **EventsProjection**

| Метод | Вход | Выход | Что делает |
|-------|------|-------|------------|
| `build_for_complex(complex_id)` | int | `EventsVM` | Собирает последние события по всем помещениям комплекса |
| `build_for_building(building_id)` | int | `EventsVM` | Собирает последние события по помещениям корпуса |
| `build_for_floor(floor_id)` | int | `EventsVM` | Собирает последние события по помещениям этажа |
| `build_for_room(room_id)` | int | `EventsVM` | Собирает последние события по помещению |

**Зависимости:** RoomRepository, EventRepository (если есть)

---

### **ListsProjection**

| Метод | Вход | Выход | Что делает |
|-------|------|-------|------------|
| `build_building_list(complex_id)` | int | `List[BuildingListItem]` | Список корпусов комплекса |
| `build_floor_list(building_id)` | int | `List[FloorListItem]` | Список этажей корпуса |
| `build_room_list(floor_id, filter)` | int, str | `List[RoomListItem]` | Список помещений этажа с фильтром по статусу |

**Зависимости:** BuildingRepository, FloorRepository, RoomRepository

## 8. ПРИНЦИПЫ РАБОТЫ

### **1. Чистота (Pure Function)**
```python
def build(self, id: int) -> ViewModel:
    # 1. Читаем данные из репозиториев
    # 2. Считаем агрегаты
    # 3. Возвращаем ViewModel
    # Никаких побочных эффектов!
```

### **2. Всегда возвращаем ViewModel**
```python
# Даже если данных нет:
def build(self, id: int) -> ContactsVM:
    if not self._building_repo.exists(id):
        return ContactsVM.empty()  # нулевые значения
```

### **3. Читаем только из репозиториев**
```python
# Правильно:
buildings = self._building_repo.get_by_complex(complex_id)

# Неправильно:
graph = self._graph.get(NodeType.BUILDING, ...)
```

### **4. Не знаем о существовании None в репозиториях**
```python
# Репозиторий кидает NotFoundError или возвращает None?
# Проекция этого не знает — она работает с тем, что есть
# Контроллер отвечает за обработку ошибок
```

## 9. РИСКИ

| Риск | Вероятность | Смягчение |
|------|-------------|-----------|
| **Проекция становится сложной** | Средняя | Делить на более мелкие методы, выносить подсчеты в утилиты |
| **Повторные вычисления** | Высокая | Контроллер может кэшировать View Models, проекция не должна |
| **Соблазн добавить кэш** | Средняя | Архитектурный запрет, code review |
| **Соблазн добавить подписки** | Низкая | Четкое разделение: проекции не знают о EventBus |
| **Слишком много обращений к репозиториям** | Средняя | Оптимизировать запросы (например, get_all_rooms_by_building) |

## 10. ВЗАИМОДЕЙСТВИЕ С ДРУГИМИ СЛОЯМИ

```
[Контроллер] вызывает проекцию
    ↓
[Проекция] читает из репозиториев (data)
    ↓
[Репозитории] читают из EntityGraph
    ↓
[Проекция] возвращает ViewModel
    ↓
[Контроллер] эмитит событие с ViewModel
    ↓
[UI] получает ViewModel и отображает
```

**Проекция НЕ знает:**
- О существовании EventBus
- О существовании контроллеров
- О существовании UI

**Проекция ЗНАЕТ:**
- О репозиториях (data)
- О View Models (view_models)
- О моделях (models)

---

## 11. ИТОГ

Проекции — это **тонкий слой преобразования**, который:
- ✅ Не имеет состояния
- ✅ Не знает о событиях
- ✅ Не принимает решений
- ✅ Только читает и преобразует
- ✅ Всегда возвращает ViewModel
- ✅ Легко тестируется

**Это позволяет:**
- UI не зависеть от формата хранения данных
- Легко менять структуру View Models без изменения data слоя
- Тестировать логику отображения изолированно
- Сохранять предсказуемость потока данных
