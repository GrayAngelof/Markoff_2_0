## Анализ слоя: **projections** (преобразование данных для UI)

### Краткое описание слоя

**Назначение** – преобразовывать данные из внутренних моделей (`models`) и репозиториев (`data`) в структуры, удобные для отображения в UI. Слой `projections` создаёт **представления** (например, узлы дерева), которые содержат только необходимую для UI информацию и методы навигации.

**Что делает:**
- Преобразует сущности (Complex, Building, Floor, Room) в узлы дерева (`TreeNode`)
- Форматирует отображаемые имена (например, «Этаж 3», «Подвал 1»)
- Добавляет информацию о наличии детей (для отображения стрелочки/плюсика)
- Сортирует дочерние узлы (по имени)
- Предоставляет структуру `TreeNode` с методами навигации по дереву (родитель, дети, поиск)

**Что не должен делать:**
- Импортировать что-либо из `controllers` или `ui` (нарушение зависимости)
- Содержать бизнес-логику (фильтрацию, валидацию, загрузку данных)
- Обращаться напрямую к `EntityGraph` или API (это задача `data` и `services`)
- Содержать логику отображения (виджеты, рендеринг)

```

```

### Внутренние классы (кратко)

| Класс | Файл | Назначение |
|-------|------|------------|
| `TreeNode` | `tree_node.py` | Узел дерева. Хранит данные (`data`), ссылки на родителя и детей, отображаемое имя, тип узла, флаг `has_children`. Предоставляет методы навигации (`append_child`, `remove_child`, `find_child_by_id`). |
| `TreeProjection` | `tree.py` | Проекция для построения дерева. Методы: `get_root_nodes()` – создаёт корневые узлы (комплексы) из репозитория; `build_children_from_payload()` – создаёт узлы для детей (корпуса, этажи, помещения) из загруженных данных. |

### Экспортируемые методы / классы для вышестоящих слоёв (`controllers`, `ui`)

Через `projections/__init__.py` экспортируются:

**Классы:**
- `TreeNode` – узел дерева (используется в `TreeModel` и `TreeView`)
- `TreeProjection` – построитель дерева

**Публичные методы `TreeProjection`:**

| Метод | Возвращает | Назначение |
|-------|------------|-------------|
| `get_root_nodes()` | `List[TreeNode]` | Создаёт корневые узлы (комплексы) из всех данных в `ComplexRepository`. Формирует отображаемое имя с количеством корпусов, если есть. |
| `build_children_from_payload(payload, child_type)` | `List[TreeNode]` | Преобразует список загруженных DTO/моделей в узлы дерева указанного типа. Сортирует узлы по имени. Родитель не устанавливается – это делает `TreeModel` при вставке. |

**Публичные методы `TreeNode` (используются в `TreeModel` и UI-обработчиках):**

| Метод / свойство | Назначение |
|----------------|------------|
| `id`, `type`, `name`, `has_children` | Свойства для отображения в виджете дерева. |
| `data` | Исходные данные (модель) – для доступа к деталям. |
| `parent` | Родительский узел. |
| `children` | Копия списка дочерних узлов. |
| `append_child(child)`, `add_children(children)` | Добавление детей. |
| `remove_child(child)`, `remove_all_children()` | Удаление детей. |
| `child_at(row)`, `child_count()` | Доступ к детям по индексу. |
| `row()` | Индекс узла в родителе. |
| `get_identifier()` | Возвращает `NodeIdentifier` для событий. |
| `find_child_by_id(node_type, node_id)` | Рекурсивный поиск узла по типу и ID. |

### Итоговое заключение: принципы работы со слоем `projections`

1. **Импорт только сверху вниз** – слой `projections` может импортировать из `core`, `models`, `data`, но **не должен** импортировать из `controllers` или `ui`. В текущем коде зависимости корректны: `tree.py` импортирует репозитории из `data` (разрешено) и `TreeNode` из своего слоя.

2. **Проекции не хранят состояние** – `TreeProjection` не имеет внутреннего состояния (кроме ссылок на репозитории, которые являются зависимостями). Все методы чисто преобразуют входные данные в узлы.

3. **Используйте `TreeProjection` для создания узлов** – не создавайте `TreeNode` вручную в контроллерах или UI. Проекция отвечает за форматирование имени, определение наличия детей и сортировку.

4. **`TreeNode` – это не модель данных** – это представление для UI. Он содержит ссылки на родителя и детей, что удобно для навигации в виджете дерева. Однако **не храните бизнес-логику** внутри `TreeNode`.

5. **Родительские связи устанавливаются снаружи** – методы `build_children_from_payload()` не устанавливают `parent` для созданных узлов. Это делает `TreeModel` (в слое `ui`) при вставке детей в конкретный родительский узел. Такой подход позволяет переиспользовать узлы.

6. **Форматирование имени** – для этажей используется специальное форматирование (отрицательные номера → «Подвал N»). Вся логика форматирования сосредоточена в `TreeProjection`, что упрощает изменение правил отображения.

7. **Сортировка** – дочерние узлы сортируются по имени (регистронезависимо) в `build_children_from_payload()`. Это единое место, гарантирующее порядок во всём дереве.

8. **Поиск узлов** – метод `find_child_by_id()` рекурсивно обходит дерево. Это удобно для синхронизации состояния (например, найти узел, который нужно раскрыть). Однако при больших деревьях следует рассмотреть индексацию.

Слой `projections` является **мостом между данными и представлением** – он преобразует «сырые» сущности в структуры, которые может эффективно отображать UI, не загрязняя бизнес-логику деталями отображения.
