## Анализ слоя: **view_models** (слой моделей представления для UI)

### Краткое описание слоя

**Назначение** – предоставлять иммутабельные контейнеры данных (View Models), которые агрегируют и структурируют информацию для отображения в пользовательском интерфейсе. Слой `view_models` является **контрактом между бизнес-логикой (сервисами/контроллерами) и UI-виджетами**, обеспечивая типобезопасную передачу данных без привязки к конкретному фреймворку.

**Что делает:**
- Определяет структуры данных для отображения статистики (комплексы, корпуса, этажи)
- Описывает модели контактов (контрагенты, контактные лица, группы)
- Предоставляет модели для списков (корпуса, этажи, помещения)
- Содержит модели для датчиков пожарной безопасности и событий
- Гарантирует иммутабельность (`frozen=True, slots=True`) для безопасного использования в UI
- Предлагает фабричные методы (`empty()`) для создания пустых состояний (fallback)

**Что не должен делать:**
- Содержать бизнес-логику, расчёты, валидацию (это в `services` или `controllers`)
- Содержать методы загрузки данных (это в `services`)
- Импортировать что-либо из `ui`, `controllers`, `services`, `data`, `projections`
- Содержать ссылки на виджеты или зависимости от Qt/других UI-фреймворков

**Зависимости:** только от стандартной библиотеки Python (`dataclasses`, `datetime`, `typing`, `field`). Никаких импортов из других слоёв приложения (даже `core` или `models` не используются). Это делает слой максимально лёгким и переиспользуемым.

### Файловая структура слоя

```
client/src/view_models/
├── __init__.py                    # Экспорт всех View Models
├── statistics.py                  # Статистика (ComplexStatisticsVM, BuildingStatisticsVM, FloorStatisticsVM, RoomTypeStat)
├── contacts.py                    # Контакты (ContactsVM, ContactGroup, ContactPerson, ContactSummary)
├── sensors.py                     # Датчики (SensorsVM, SensorIssue)
├── events.py                      # События (EventsVM, EventItem)
└── lists.py                       # Элементы списков (BuildingListItem, FloorListItem, RoomListItem)
```

### Внутренние классы (кратко)

| Класс | Модуль | Назначение |
|-------|--------|------------|
| `ComplexStatisticsVM` | `statistics.py` | Статистика для комплекса: количество корпусов/этажей/помещений, площади, занятость, разбивка по типам помещений. |
| `BuildingStatisticsVM` | `statistics.py` | Статистика для корпуса: этажи, помещения, площади, занятость. |
| `FloorStatisticsVM` | `statistics.py` | Статистика для этажа: помещения, занятость, разбивка по типам. |
| `RoomTypeStat` | `statistics.py` | Статистика по одному типу помещения (название, количество, площадь). |
| `ContactsVM` | `contacts.py` | Агрегированные данные по контрагентам: количество организаций, арендаторов, собственников, должников; группы контактов; сводка по площадям и долгам. |
| `ContactGroup` | `contacts.py` | Группа контактов с определённой категорией (например, "Юридические", "Финансовые"). |
| `ContactPerson` | `contacts.py` | Отдельное контактное лицо: имя, должность, телефон, email, флаг приоритетности. |
| `ContactSummary` | `contacts.py` | Сводка по контрагентам: общая площадь, сданная площадь, процент занятости, общая задолженность. |
| `SensorsVM` | `sensors.py` | Датчики: общее количество, активные, неактивные, на обслуживании; список проблемных датчиков; списки ID для кликабельности. |
| `SensorIssue` | `sensors.py` | Проблемный датчик: ID, локация, описание проблемы, дата последней проверки. |
| `EventsVM` | `events.py` | События: общее количество, список недавних событий, флаг "показать ссылку на все события". |
| `EventItem` | `events.py` | Отдельное событие: timestamp, тип, локация, описание, критичность. |
| `BuildingListItem` | `lists.py` | Элемент списка корпусов: ID, название, количество этажей. |
| `FloorListItem` | `lists.py` | Элемент списка этажей: ID, номер, количество помещений. |
| `RoomListItem` | `lists.py` | Элемент списка помещений: ID, номер, площадь, статус. |

### Внутренние импорты

**Импортов из других слоёв приложения (core, models, data, services, projections, controllers, ui) нет.**  

Все импорты – только из стандартной библиотеки:
- `dataclasses` (dataclass, field)
- `datetime` (datetime)
- `typing` (List, Optional)
- Внутри одного модуля (например, `contacts.py` импортирует свои же классы)

**Никаких импортов из `utils.logger` или других служебных модулей.**

### Экспортируемые методы / классы для вышестоящих слоёв (controllers, ui)

Публичное API слоя `view_models` доступно через `from src.view_models import ...` (согласно `view_models/__init__.py`):

**Статистика:**
- `ComplexStatisticsVM`
- `BuildingStatisticsVM`
- `FloorStatisticsVM`
- `RoomTypeStat`

**Контакты:**
- `ContactsVM`
- `ContactGroup`
- `ContactPerson`
- `ContactSummary`

**Датчики:**
- `SensorsVM`
- `SensorIssue`

**События:**
- `EventsVM`
- `EventItem`

**Списки:**
- `BuildingListItem`
- `FloorListItem`
- `RoomListItem`

**Методы каждого класса:**
- Конструктор со значениями по умолчанию (все поля опциональны, имеют значения по умолчанию)
- `empty()` – классовый метод, возвращающий экземпляр со значениями по умолчанию (для fallback)
- Все поля доступны для чтения (иммутабельные атрибуты)

### Итоговое заключение: принципы работы со слоем `view_models`

1. **Чистый слой данных, не зависящий от приложения** – `view_models` не импортирует ничего, кроме стандартной библиотеки. Это позволяет использовать их в любом контексте (даже в тестах без поднятия окружения).

2. **Иммутабельность обязательна** – все классы помечены `@dataclass(frozen=True, slots=True)`. Это гарантирует, что UI не сможет случайно изменить данные, и упрощает отслеживание изменений (например, при сравнении старой и новой VM).

3. **Никакой логики, только данные** – в этих классах нет методов, вычисляющих значения (кроме `empty`). Все расчёты (процент занятости, суммарная площадь) должны выполняться в сервисах или контроллерах, а готовые результаты передаваться в VM.

4. **Использование фабрики `empty()`** – для безопасного отображения состояния "нет данных" или при ошибке загрузки UI может вызвать `ComplexStatisticsVM.empty()`, что даст предсказуемую структуру с нулевыми значениями.

5. **Связь с `models` и `data`** – хотя прямых импортов нет, View Models часто создаются на основе DTO из `models` и агрегируют данные из нескольких источников. Преобразование DTO → VM должно происходить в `services` или `controllers`, но не в `ui`.

6. **Разделение по темам** – файлы сгруппированы по функциональным вкладкам UI: `statistics` (Физика), `contacts` (Юрики), `sensors` и `events` (Пожарка), `lists` (общие списки). При добавлении новой вкладки создавайте новый файл и экспортируйте VM в `__init__.py`.

7. **Использование `field(default_factory=list)`** – для коллекций, чтобы избежать мутабельных значений по умолчанию (например, `groups: List[ContactGroup] = field(default_factory=list)`).

8. **Кликабельные ID** – в `SensorsVM` есть поля `active_ids`, `inactive_ids`, `maintenance_ids`, которые позволяют UI при клике на статистику запросить список соответствующих датчиков. Это пример того, как VM может содержать не только отображаемые данные, но и служебные идентификаторы для навигации.

Слой `view_models` является **последним звеном перед UI** – он обеспечивает типовую безопасность и иммутабельность данных, которые виджеты будут отображать. Благодаря отсутствию зависимостей, эти модели можно легко сериализовать, кэшировать или передавать между потоками.
